# Component Review Checklist

> Objetivo: asegurar que un componente no solo funcione, sino que cumpla los estándares de calidad del proyecto.

---

# 1. Validación visual y funcional

## Pregunta

**¿Este componente resuelve correctamente el problema para el que fue diseñado?**

### Diseño

* [ ] Es fiel al diseño aprobado.
* [ ] Mantiene la jerarquía visual esperada.
* [ ] Respeta el Design System.

### Responsive

* [ ] Se visualiza correctamente en mobile.
* [ ] Se visualiza correctamente en tablet.
* [ ] Se visualiza correctamente en desktop.

### Estados

* [ ] Hover (si aplica)
* [ ] Focus (si aplica)
* [ ] Active (si aplica)
* [ ] Disabled (si aplica)
* [ ] Loading (si aplica)
* [ ] Error (si aplica)
* [ ] Estado vacío (si aplica)

### Compatibilidad

* [ ] Funciona correctamente en los navegadores soportados por el proyecto.

---

# 2. HTML

## Pregunta

**¿La estructura comunica correctamente el significado del contenido?**

* [ ] Usa HTML semántico.
* [ ] La jerarquía es correcta.
* [ ] No existen elementos innecesarios.
* [ ] La estructura es clara y fácil de entender.

---

# 3. Accesibilidad

## Pregunta

**¿El componente puede ser utilizado correctamente por la mayor cantidad de usuarios posible?**

* [ ] Es navegable con teclado cuando aplica.
* [ ] Tiene una estructura lógica de lectura.
* [ ] Los elementos interactivos son identificables.
* [ ] Los estados e interacciones cumplen con las buenas prácticas de accesibilidad del proyecto.
